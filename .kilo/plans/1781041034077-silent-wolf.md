# خطة تكامل Recommendations & Matching بين StayMatch و ASP.NET

## Context

- ASP.NET API هو الـ gateway الرئيسي — الـ Frontend بيتكلم معاه بس
- StayMatch (FastAPI + PostgreSQL/Neon) هو المسؤول عن recommendations + matching
- الـ user_id هو نفس الـ GUID عبر النظامين — مفيش حاجة mapping
- الـ 16 سؤال بتاعت StayMatch هي الـ source of truth للـ questionnaire

---

## 1. Data Sync (MSSQL → PostgreSQL)

الحالي شغال: `Properties`, `Rooms`, `PropertyAmenities`, `AllowedTenants` بتت sync.

**المطلوب إضافته:**

| المصدر (MSSQL) | الهدف (PostgreSQL) | التوقيت |
|---|---|---|
| `AspNetUsers.Id` + `AspNetUsers.*` | `user_profiles` | أول sync + كل ما user يتغير |
| `AspNetUsers.Id` + `UserPreferences` (لو موجود) | `user_search_preferences` | أول sync |

- محتاجين `Users` table sync أو `POST /users/profile` من ASP.NET بعد التسجيل
- ASP.NET يركب `POST /users/profile` بعد كل register/login

**ملاحظة:**
الـ staymatch دلوقتي بيقرأ chatbot `user_preferences` table في `PreferencesBridge`. محتاج نضيف source تاني (إما MSSQL Users table أو API call من ASP.NET).

---

## 2. Questionnaire

StayMatch عنده 16 سؤال في 4 categories:

1. Personality & Career (3 questions)
2. Lifestyle & Habits (5 questions)
3. Social & Cohabitation (4 questions)
4. Finance & Cleanliness (4 questions)

**الخطة:**

1. تشغيل `python scripts/seed_questionnaire.py` (مرة واحدة) لزرع الـ 16 سؤال في PostgreSQL
2. ASP.NET يستخدم `GET /questionnaire/questions` عشان يجيب الأسئلة ويعرضها للمستخدم
3. لما المستخدم يsubmit answers:
   - ASP.NET يبعت `POST /questionnaire/answers/{user_id}`
   - StayMatch بيحفظ الإجابات + يشغل background recomputation للـ recommendations و matching
4. ASP.NET يقدر يجيب إجابات المستخدم بأي وقت عن طريق `GET /questionnaire/answers/{user_id}`

**ASP.NET side:** إلغاء `/api/Question` و `/api/Answer` endpoints القديمة واستخدام StayMatch APIs.

## 3. Recommendation Flow

### Architecture

```
Frontend → ASP.NET → (internal HTTP) → StayMatch FastAPI
                                      ↓
                                 PostgreSQL (Neon)
                                      ↓
                                 scored result → JSON → ASP.NET → Frontend
```

### Pre-computation Strategy (للـ High Performance <100ms)

> الـ compute الثقيل (scoring كل الشقق ضدユーザ preferences) يتعمل في background.

| السيناريو | الإجراء |
|---|---|
| المستخدم submit questionnaire answers | `POST /questionnaire/answers/{user_id}` → auto background recompute |
| المستخدم يغير preferences | ASP.NET يبعت `POST /users/preferences` + `POST /recommend/compute/{user_id}` |
| المستخدم يعمل first login | `POST /recommend/compute/{user_id}` (أول مرة) |
| المستخدم بيطلب recommendations (cold) | `GET /recommend/properties/{user_id}` → يرجع cached result (<10ms) |
| Admin يغير weights | `PUT /admin/weights/{group}/{key}?value=X` → next recompute يستخدم الوزن الجديد |

### GET /recommend/properties/{user_id}

**الوضع الحالي:**
- بيفحص cache (1h in-memory, 24h DB)
- لو مفيش cache → بي compute كل حاجة live
- لو مفيش user ولا preferences → بيرجع popular properties

**تحسينات مطلوبة:**
- إضافة query parameters للفلترة السريعة:
  - `?city=Cairo&min_budget=2000&max_budget=5000&property_type=shared`
  - الفلترة دي تتطبق على cached results (مش محتاج recompute)
- إضافة `?limit=10` للتحكم في عدد النتائج
- ASP.NET ممكن يضيف filter params عشان يظبط الليسته

**Response شكلها:**
```json
{
  "user_id": "guid-...",
  "recommendations": [
    {
      "property_id": 42,
      "score": 0.87,
      "score_breakdown": {
        "budget": 1.0, "location": 1.0, "amenities": 0.67,
        "tenant": 1.0, "furnished": 1.0, "property_type": 1.0, "recency": 0.5
      },
      "rank": 0
    }
  ]
}
```

ASP.NET بعد ما يستلم الـ response:
- يضيف `images` (من الـ `PropertyImages` table في MSSQL/ASP.NET DB)
- يضيف `host_name`, `host_avatar` (من `AspNetUsers`)
- ممكن يضيف `distance_from_user` لو عاوز

### GET /recommend/rooms/{user_id}

نفس الفكرة بس للغرف.
- Diversity rule: max 2 rooms per building
- Response بيضم `property_id` عشان ASP.NET يقدر يربط.

### POST /recommend/compute/{user_id}

- بي recompute property + room recommendations في background task
- بيرجع `{"status": "ok", "message": "Recomputing..."}`
- ASP.NET ميستنيش — يبعت الـ request ويخلص

## 4. Matching Flow

### Architecture

```
ASP.NET → POST /match/compute/{user_id}
       ← {"status":"completed","matches":[{room_id, property_id, room_compatibility_score, roommate_count}]}
```

### خطوات الـ compute:

1. جلب إجابات seeker من `user_questionnaire_answers`
2. جلب كل users التانيين اللي عندهم إجابات
3. جلب كل rooms اللي عليها capacity متاح
4. لكل room:
   - لكل other user: compute pairwise score (questionnaire 50% + gender 15% + occupation 10% + age 10% + lifestyle 15%)
   - Filter scores >= 0.3
   - Aggregate room score = avg(pairwise) * capacity_factor
5. حفظ في `roommate_matches` table (مع `match_breakdown` الـ secret)
6. رجع aggregate scores بس (مترجعش `match_breakdown` أو `pairwise_scores`)

### GET /match/results/{user_id}

- بيرجع stored matches sorted by score desc
- ASP.NET بيستخدمها عشان يعرض compatibility scores

### تحسينات Performance:

- الـ compute ممكن يكون تقيل لو users كتير → `O(users × rooms)`
- الحل: Pre-compute بعد ما seeker يSubmit questionnaire (مع background task)
- Cache: الـ `roommate_matches` عنده 24h TTL
- ممكن نضيف incremental matching: بس users اللي اتغيرت إجاباتهم من آخر مرة

## 5. Interactions & Feedback Loop

### User Clicks on Property

```
User views property #42 in frontend
→ Frontend sends event to ASP.NET
→ ASP.NET calls POST /interactions
   Body: {user_id, target_type:"property", target_id:42, action:"viewed", dwell_seconds: 35, search_lat, search_lng}
→ StayMatch logs to user_interactions table
→ Next GET /recommend uses interaction history for boosting
```

### Actions:

| Frontend Action | `action` value | `dwell_seconds` | Effect |
|---|---|---|---|
| Saw property in list | `"viewed"` | 0 | يعتبر passive interest |
| Opened property details | `"viewed"` | ثواني اللي قعد عليها | 30s+ = strong interest |
| Saved property | `"saved"` | null = قوي | boost score |
| Liked property | `"liked"` | null | boost score |
| Contacted owner | `"contacted"` | null | highest boost |
| Skipped/swiped away | `"skipped"` | 0 | slight penalty |

### Periodic Analysis

ASP.NET ممكن يبعت `POST /analyze/{user_id}`:
- بعد 5+ interactions
- بيحلل dwell time, saves, likes
- بي inferences الـ preferences ويسجلها في `user_search_preferences`
- بيأثر على الـ recommendations الجاية

## 6. Caching Strategy (High Performance)

### المستوى الأول: In-Memory (staymatch process)

- TTL: 1 hour
- شغال دلوقتي في `_check_cache()`
- الصعوبة: لو service اتrestart، cache بيروح

### المستوى التاني: PostgreSQL (Database)

- `property_recommendations` table: 24h TTL
- `room_recommendations` table: 24h TTL
- `roommate_matches` table: 24h TTL
- بيمسح القديم و يكتب الجديد مع كل recompute

### المستوى التالت (مستقبلي): Redis

- ممكن لو load زاد
- TTL: 1 hour
- أسرع من PostgreSQL

### Pre-compute Strategy للـ <100ms:

```
User registers / logs in
  → POST /recommend/compute/{user_id}  (background, slow ~2-5s)
  → Done. Cached.

User browses properties
  → GET /recommend/properties/{user_id}  (instant, returns cache <10ms)
  → Done.

User submits questionnaire
  → POST /questionnaire/answers/{user_id} (background recompute triggered)
  → POST /recommend/compute/{user_id} auto-runs in background
  → Done. Cached.
```

## 7. Deployment

### Railway (Docker) — زي ما هو:

```
Dockerfile:
  CMD alembic upgrade head && uvicorn app.main:app --host 0.0.0.0 --port 8000
```

### Environment Variables:

```
DATABASE_URL=postgresql://...?sslmode=require       # Neon PostgreSQL
MSSQL_CONNECTION_STRING=DRIVER={ODBC Driver 17}...   # ASP.NET MSSQL
SYNC_INTERVAL_MINUTES=5                               # Auto sync period
API_KEY=staymatch-rec-api-key-2026                    # ASP.NET يستخدمها في الـ header
LOG_LEVEL=INFO
```

### Railway و ASP.NET في نفس البيئة:

- ASP.NET عنده service URL بتاع StayMatch (مثلاً `http://staymatch:8000`)
- بيبعت requests مع `X-API-Key` header
